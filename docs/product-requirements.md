# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**Retail Sales Forecasting Pipeline** - 小売・マーケティング向け週次売上予測パイプライン

### プロダクトコンセプト
- **再利用可能な予測テンプレート**: Notebook中心の学習ログではなく、CLI、設定ファイル、テスト、レポートを備えた実務再利用可能なPythonパッケージとして構成する。
- **マーケティング要因を含むモデル比較**: 売上実績だけでなく、デジタル広告、オフライン広告、TVCM、印刷媒体などの外部要因を組み込み、複数モデルを同一条件で評価する。
- **ポートフォリオとして伝わる成果物**: Python、時系列予測、機械学習、マーケティング分析、MLOps的な設計力をREADME、レポート、ドキュメントで確認できる状態にする。

### プロダクトビジョン
小売・マーケティング領域の分析担当者が、週次売上と広告施策データを使って、再現性のある売上予測を素早く実行できるパイプラインを提供する。複数の統計モデル・機械学習モデルを共通インターフェースで扱い、Baselineを上回る価値があるかを定量的に判断できるようにする。最終的には、CLI実行からモデル評価、将来予測、ビジネス向けレポート生成までを一貫して実行できる構成を目指す。

### 目的
- 小売・マーケティング領域における週次売上予測パイプラインを構築する。
- 複数モデルを同一条件で比較し、Baseline比の改善を評価できるようにする。
- CLIでデータ検証、学習、評価、予測、レポート生成を再現可能にする。
- ポートフォリオとして、Python、時系列予測、機械学習、マーケティング分析、設計力を示す。
- Codex、Antigravity、CursorなどのAIコーディングエージェントが拡張しやすい構造にする。

## ターゲットユーザー

### プライマリーペルソナ: マーケティング分析担当者
- SEO、Paid Media、CRM、販売計画などの業務で週次売上や広告施策の効果を確認している。
- CSV、Excel、BIツール、Python Notebookを利用するが、再現可能な予測パイプラインの整備には課題がある。
- モデル精度だけでなく、広告施策や販促計画に使えるビジネス示唆を必要としている。
- CLIまたは設定ファイルで、同じ条件のモデル比較と将来12週間の予測を再実行したい。
- 典型的なワークフロー: 週次データ更新、データ検証、モデル評価、予測結果確認、関係者向けレポート作成。

### セカンダリーペルソナ: データアナリスト / Python学習者
- pandas、scikit-learn、LightGBM、Prophet、statsmodelsなどを使った時系列予測を学びたい。
- Notebookで試したモデルをproduction codeへ整理する方法を確認したい。
- テスト、設定ファイル、CLI、レポート生成を含む実務寄りのPythonプロジェクト構成を参考にしたい。

### ポートフォリオ閲覧者
- 採用担当者、面接官、データ分析・マーケティング領域の技術評価者。
- 短時間でプロジェクトの目的、実装範囲、技術選定、成果物、拡張性を確認したい。

## 成功指標(KPI)

### プライマリーKPI
- CLI再現性: サンプルデータで `validate-data`、`train`、`evaluate`、`forecast`、`report` がすべて正常終了する。
- モデル比較の成立: Baseline Modelを含む2種類以上のモデル評価結果をCSVまたはJSONで出力できる。
- 将来予測の成立: `Record_Type=Forecast` の外部要因を用いて12週間先の予測CSVを生成できる。
- レポート生成の成立: モデル比較、予測グラフ、誤差分析、ビジネス示唆を含むHTMLまたはMarkdownレポートを生成できる。
- テスト品質: 主要処理の単体テストが `pytest` で通る。

### セカンダリーKPI
- README理解容易性: READMEを読むだけで、目的、主要機能、セットアップ、実行方法、サンプル出力、データの扱いが分かる。
- 拡張容易性: 新規モデルを追加する際の実装先、共通インターフェース、設定追加、テスト追加のルールが明記されている。
- データ安全性: 実在企業の機密データを含まず、サンプルデータまたはダミーデータのみで再現できる。

## 機能要件

### コア機能(MVP)

#### データ読み込み

**ユーザーストーリー**:
分析担当者として、週次売上と外部要因を予測モデルに渡すために、CSVを標準形式で読み込みたい。

**受け入れ条件**:
- [ ] `data/sales_data.csv` のようなCSVファイルを読み込める。
- [ ] 必須カラム `Date`、`Sales`、`Record_Type` が存在しない場合はエラーを返す。
- [ ] `TVCM_GPR`、`Print_Media`、`Offline_Ads`、`Digital_Ads` を外部要因として扱える。
- [ ] カンマ区切りの数値文字列を数値型へ変換できる。
- [ ] 読み込み後に内部標準カラム名へ変換できる。

**優先度**: P0(必須)

#### データバリデーション

**ユーザーストーリー**:
分析担当者として、モデル学習前のエラーを早期に見つけるために、入力データの形式と欠損を検証したい。

**受け入れ条件**:
- [ ] 日付形式、数値変換可否、必須カラム、重複週、欠落週を検証できる。
- [ ] `Record_Type` が `Actual` または `Forecast` であることを検証できる。
- [ ] `Actual` 行では `Sales` の存在を必須とする。
- [ ] `Forecast` 行では `Sales` 欠損を許容し、外部要因の存在を確認する。
- [ ] 検証結果をCLI上に分かりやすく表示できる。

**優先度**: P0(必須)

#### 前処理・特徴量エンジニアリング

**ユーザーストーリー**:
データアナリストとして、複数モデルで使える学習データを作るために、時系列特徴量とマーケティング特徴量を一貫して生成したい。

**受け入れ条件**:
- [ ] 日付順ソート、欠損処理、型変換、カテゴリ変数処理ができる。
- [ ] train / validation / testを時系列順に分割できる。
- [ ] ラグ特徴量として前週、4週前、12週前、52週前売上を生成できる。
- [ ] 4週、8週、12週移動平均を生成できる。
- [ ] 年、月、週番号、四半期、年末年始フラグを生成できる。
- [ ] 広告費、TVCM GPR、印刷媒体費などの外部変数を特徴量として扱える。

**優先度**: P0(必須)

#### Baseline Model

**ユーザーストーリー**:
分析担当者として、高度なモデルが本当に価値を出しているか判断するために、単純なベースライン予測と比較したい。

**受け入れ条件**:
- [ ] Naive Forecastとして前週売上を予測値にできる。
- [ ] Seasonal Naive Forecastとして前年同週売上を予測値にできる。
- [ ] Moving Average Forecastとして過去4週平均を予測値にできる。
- [ ] Baseline Modelも共通インターフェース `fit(train_data)`、`predict(future_data)` を持つ。

**優先度**: P0(必須)

#### モデル学習

**ユーザーストーリー**:
データアナリストとして、複数モデルを同じ手順で評価するために、共通インターフェースでモデルを学習したい。

**受け入れ条件**:
- [ ] 初期対象モデルとして Baseline、Linear Regression、LightGBM、Prophet、SARIMAX を扱える設計にする。
- [ ] 各モデルは `fit(train_data)`、`predict(future_data)` を実装する。
- [ ] 設定ファイルから利用モデルを選択できる。
- [ ] モデルごとのパラメータを設定ファイルで管理できる。
- [ ] 学習済みモデルを `outputs/models/` に保存できる。

**優先度**: P1(重要)

#### モデル評価・時系列検証

**ユーザーストーリー**:
分析担当者として、実務利用できる精度か判断するために、時系列に配慮した検証と複数指標での比較を行いたい。

**受け入れ条件**:
- [ ] RMSE、MAE、MAPE、sMAPE、WAPE、Bias、Baseline比改善率を算出できる。
- [ ] MAPE、sMAPE、WAPE、Baseline比改善率のゼロ除算・欠損ケースの扱いを定義し、テストできる。
- [ ] モデル別の評価結果を表形式で出力できる。
- [ ] 評価結果をCSVまたはJSONで保存できる。
- [ ] train / validation / testを時系列順に分割できる。
- [ ] walk-forward validationを実装し、各検証期間ごとのスコアを保存できる。
- [ ] 実績値と予測値のグラフを生成し、誤差の大きい週を特定できる。

**優先度**: P0(必須)

#### 将来予測

**ユーザーストーリー**:
販売計画担当者として、今後の売上見込みを確認するために、将来12週間の週次売上を予測したい。

**受け入れ条件**:
- [ ] デフォルト予測期間は12週間とする。
- [ ] `Record_Type=Forecast` の行を将来予測用の外部要因として利用できる。
- [ ] `--horizon` で予測期間を指定できる。
- [ ] 予測結果をCSVで出力できる。
- [ ] 実績値と将来予測を連続したグラフで表示できる。

**優先度**: P0(必須)

#### レポート生成

**ユーザーストーリー**:
マーケティング分析担当者として、関係者に結果を説明するために、モデル評価とビジネス示唆を含むレポートを自動生成したい。

**受け入れ条件**:
- [ ] HTMLまたはMarkdown形式のレポートを生成できる。
- [ ] Executive Summary、Dataset Overview、Forecasting Approach、Model Comparisonを含む。
- [ ] Best Model Result、Error Analysis、Feature Importance、12-week Forecastを含む。
- [ ] Business Insights、Limitations、Next Stepsを含む。
- [ ] `--output` で出力先を指定できる。

**優先度**: P1(重要)

#### CLI

**ユーザーストーリー**:
開発者として、Notebookを開かずに処理を再現するために、主要処理をCLIから実行したい。

**受け入れ条件**:
- [ ] `python -m sales_forecast validate-data` を実行できる。
- [ ] `python -m sales_forecast train` を実行できる。
- [ ] `python -m sales_forecast evaluate` を実行できる。
- [ ] `python -m sales_forecast forecast` を実行できる。
- [ ] `python -m sales_forecast report` を実行できる。
- [ ] `--config`、`--model`、`--horizon`、`--output` を用途に応じて指定できる。
- [ ] エラー時に原因が分かるメッセージを表示する。

**優先度**: P0(必須)

#### 設定ファイル管理

**ユーザーストーリー**:
開発者として、実行条件をコード変更なしで切り替えるために、データパス、モデル、評価条件をYAMLで管理したい。

**受け入れ条件**:
- [ ] `configs/base.yaml` を標準設定ファイルとして利用できる。
- [ ] データパス、日付列、目的変数、レコード種別列、外部変数を設定できる。
- [ ] 学習期間、検証期間、テスト期間、利用モデル、モデルパラメータを設定できる。
- [ ] 評価指標とレポート出力先を設定できる。

**優先度**: P0(必須)

#### テスト

**ユーザーストーリー**:
開発者として、AIエージェントや人間が安全に拡張できるように、主要処理の単体テストを実行したい。

**受け入れ条件**:
- [ ] `pytest` でテストを実行できる。
- [ ] データ読み込み、バリデーション、特徴量生成、評価指標、Baseline Modelをテストする。
- [ ] CLIの基本動作をテストする。
- [ ] サンプルデータでテストが通る。

**優先度**: P0(必須)

#### AIエージェント向け開発ルール

**ユーザーストーリー**:
AIコーディングエージェントを使う開発者として、意図しない編集を避けるために、プロジェクト構造と開発ルールを明文化したい。

**受け入れ条件**:
- [ ] `AGENTS.md` を作成する。
- [ ] Project overview、Directory structure、Development rulesを含める。
- [ ] Model interface、Testing rules、Data privacy rulesを含める。
- [ ] 新規モデル追加とレポート更新の手順を含める。
- [ ] Notebookは探索用、production codeは `src/` 配下に置くルールを明記する。

**優先度**: P1(重要)

### CLIインターフェース

```bash
python -m sales_forecast validate-data --config configs/base.yaml
python -m sales_forecast train --config configs/base.yaml --model lightgbm
python -m sales_forecast evaluate --config configs/base.yaml
python -m sales_forecast forecast --config configs/base.yaml --model lightgbm --horizon 12
python -m sales_forecast report --config configs/base.yaml --output reports/model_comparison_report.md
```

## 非機能要件

### 再現性
- サンプルデータを使って誰でも同じCLI手順を再現できる。
- READMEにセットアップ、データ検証、学習、評価、予測、レポート生成の手順を明記する。
- 依存ライブラリは `pyproject.toml` または `requirements.txt` で管理する。

### 保守性
- モデルごとの処理を `src/sales_forecast/models/` 配下に分離する。
- 共通処理を重複させず、データ読み込み、特徴量生成、評価指標、レポート生成を責務ごとに分割する。
- Notebookは探索用に限定し、production codeは `src/` 配下へ配置する。

### 拡張性
- 新しいモデルは `BaseForecastModel` を継承し、`fit()` と `predict()` を実装するだけで追加できる設計にする。
- 新しい評価指標、外部変数、レポート項目を追加しやすい構成にする。
- 将来的なStreamlit化、API化に備えてCLIとコアロジックを分離する。

### ユーザビリティ
- CLIエラー時は、ファイル未存在、必須カラム不足、日付不正、欠損など原因が分かるメッセージを表示する。
- READMEの冒頭でプロジェクトの価値、主要機能、実行例が短時間で理解できる。
- サンプルレポートをGitHub上で確認できる状態にする。

### 信頼性
- 時系列分割とwalk-forward validationにより、データリークを避ける。
- 評価指標は共通評価機能で算出し、モデルクラス内に重複実装しない。
- テスト対象は、データ読み込み、検証、特徴量生成、評価指標、Baseline Model、CLI基本動作とする。

### セキュリティ・データプライバシー
- 実在企業の機密データは使用しない。
- サンプルデータはダミーデータまたは公開可能なデータのみとする。
- READMEにデータの扱いを明記する。
- `.gitignore` でraw dataやprivate dataを除外する。

## スコープ外

初期スコープでは以下を対象外とする。

- 本番環境への自動デプロイ
- リアルタイム予測API
- データベース連携
- 本格的なダッシュボードアプリ開発
- 機密性の高い実データの利用
- 本番データを使った自動モデル再学習のスケジューリング
- 大規模MLOps基盤の構築

## 開発フェーズ

### Phase 1: プロジェクト再構成
- `src/`、`configs/`、`data/`、`docs/`（`docs/notebooks/` を含む）、`tests/` を整理する。
- 既存Notebookを保持しつつ、今後の実装先を明確にする。

### Phase 2: CLIとBaseline Model
- CLI、データ読み込み、データバリデーション、特徴量生成、Baseline Model、評価指標を実装する。
- `validate-data`、`train`、`evaluate` が動作する状態にする。
- 重い依存を避け、標準ライブラリ、pandas、numpy、scikit-learn中心で最小パイプラインを成立させる。

### Phase 3: 複数モデル対応
- Linear Regression、LightGBM、Prophet、SARIMAXを共通インターフェースで実装する。
- モデル比較表とBaseline比改善率を出力する。
- LightGBM、Prophet、statsmodelsはoptional dependencyとして扱える構成を検討する。

### Phase 4: 将来予測
- 12週間予測、`Record_Type=Forecast` 行の外部要因利用、予測CSVとグラフ出力を実装する。

### Phase 5: レポート生成とポートフォリオ化
- HTMLレポート、グラフ、サンプルレポート、README、docs、AGENTS.mdを整備する。

## リスクと対応策

### 実データが使えない
- ダミーデータまたは公開可能なサンプルデータを作成する。
- データ生成スクリプトを用意する。
- READMEにサンプルデータであることを明記する。

### モデルが複雑になりすぎる
- Phase 2ではBaselineを優先し、複雑なモデルはPhase 3以降に実装する。
- 共通インターフェースと評価機能を先に定義する。

### レポートが技術評価に偏る
- Executive SummaryとBusiness Insightsを必須項目にする。
- モデル精度だけでなく、意思決定への使い方と制約を説明する。

### AIエージェントでの改修が不安定になる
- `AGENTS.md` を用意し、ディレクトリ構成、編集対象、テスト、データプライバシーのルールを明記する。
- Notebookとproduction codeの役割を分ける。
